## Critical Issues

### Issue 1: Type Mismatch - last_four vs masked_key (BLOCKING)

**Severity:** 🔴 **HIGH** - Blocks proper key list display

**Location:**
- Frontend type: `frontend/src/types/apiKey.ts:4`
- Frontend usage: `frontend/src/components/api-keys/ApiKeyList.tsx:78`
- Backend endpoint: `backend/app/api/endpoints/api_keys.py:118`

**Problem:**
```typescript
// Frontend expects:
interface ApiKey {
  last_four: string  // ❌ Backend doesn't return this from list endpoint
}

// Backend returns:
{
  "masked_key": "och_live_...abcd"  // ✅ Returns full masked format
  // ❌ Does NOT return last_four field
}

// Frontend renders:
<span>och_live_****{key.last_four}</span>  // → "och_live_****undefined"
```

**Root Cause:**
- Backend model has `last_four` column
- Model's `to_dict()` only includes `last_four` if `include_sensitive=True`
- List endpoint uses default serialization without `last_four`
- Frontend type definition expects `last_four` to always be present

**Impact:**
- Key list displays `och_live_****undefined` for all keys
- Users cannot see which key is which
- Breaks REQ-F-010 (Masked Key Display)

**Fix Options:**

**Option A (Recommended - Simpler):** Use masked_key in frontend
```typescript
// Update frontend/src/types/apiKey.ts
interface ApiKey {
  masked_key: string  // Use what backend already provides
}

// Update frontend/src/components/api-keys/ApiKeyList.tsx:78
<span>{key.masked_key}</span>  // Display full masked key
```

**Option B:** Add last_four to backend response
```python
# Update backend/app/api/endpoints/api_keys.py:118
return {
    "keys": [
        {
            **key.to_dict(),
            "last_four": key.last_four,  # Explicitly add
        }
        for key in keys
    ]
}
```

**Recommendation:** Option A (use masked_key) is simpler and matches backend's existing pattern. The `masked_key` property already provides the full display format (`och_live_****abcd`).
