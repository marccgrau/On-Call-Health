### Flow 4: API Key Secret Management
**Status:** ⚠️ BROKEN

**Break Point:** ECS task definition lacks Secrets Manager configuration

**Issue:** Task definition at `infra/ecs/task-definition.json` has environment variable for `ONCALLHEALTH_API_URL` but **no secret configuration for ONCALLHEALTH_API_KEY**.

**Expected:**
```json
"secrets": [
  {
    "name": "ONCALLHEALTH_API_KEY",
    "valueFrom": "arn:aws:secretsmanager:REGION:ACCOUNT_ID:secret:mcp/api-key"
  }
]
```

**Impact:** Deployed container at http://13.218.46.36:8080 is running but MCP tools would fail with "Missing API key" error when invoked. Health check works (bypasses auth) but actual tool calls require API key.

**Recommendation:** Add secrets configuration to task-definition.json OR document that API key must be provided by client via X-API-Key header (not server-side injection).

## Tech Debt

### Non-Critical Issues

1. **API Key Injection Model Ambiguous**
   - Current implementation: Tools extract API key from X-API-Key header (client-provided)
   - Deploy script mentions: Secrets Manager secret creation (server-side)
   - **Clarification needed:** Is API key client-provided (header) or server-provided (secret)?

2. **Auth Pattern Divergence**
   - Backend uses: `context.py` with `get_api_key()` (ContextVar-based)
   - PyPI uses: `auth.py` with `extract_api_key_header(ctx)` (FastMCP Context parameter)
   - Both work but different patterns for same purpose
   - **Impact:** Maintenance burden, harder to keep in sync

3. **REQUIREMENTS.md Out of Sync**
   - AWS-01 through AWS-05 marked "Pending" but actually implemented
   - Should be updated to "Complete" with evidence from 11-01-SUMMARY.md
