### Requirement Adjustments

**REQ-F-010: Masked Key Display**

**Original specification:**
- System shall display only the last 4 characters of API keys after creation
- Format: `och_live_****1234`

**Adjusted implementation:**
- Backend returns full `masked_key` field in format `och_live_...abcd`
- Frontend displays `masked_key` directly (not constructing from `last_four`)
- **Reason:** Backend chose to compute and return full masked format for consistency across all API responses
- **Impact:** Frontend type mismatch initially caused undefined display, fixed in commit 1d14ff73
- **Outcome:** Requirement satisfied with adjusted implementation approach

## Out of Scope (Explicitly Deferred)

These requirements were intentionally excluded from v1.0:

**OOS-001:** REST API for Key Management - Security (prevents compromised key escalation)
**OOS-002:** IP Address Restrictions - High complexity, wait for demand
**OOS-003:** Audit Logging - Can add when compliance requirements emerge
**OOS-004:** Request Count Tracking - Can add when usage insights requested
**OOS-005:** Granular Permission Scopes - Ship v1 faster, add read-only/write-only in v2
**OOS-006:** Test Button in UI - Creates rate limit edge cases, docs sufficient
**OOS-007:** Email Notifications for Expiry - Infrastructure complexity, spam concerns
**OOS-008:** Automatic Key Rotation - Breaks integrations silently
**OOS-009:** Retrievable Keys - Security principle (never store plaintext)
